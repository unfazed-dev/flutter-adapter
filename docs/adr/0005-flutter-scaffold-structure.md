# ADR-0005: Flutter Scaffold Template Structure

**Status:** Proposed
**Date:** 2026-05-27

## Context

Per the fat-adapter model (kit ADR-0001), an adapter ships scaffold templates
that generate initial project structure. These templates use Cookiecutter-style
variable substitution (`{{ variable }}`) and are resolved at `maestro kit init`
from defaults.toml + the project brief.

**Variable substitution**: Cookiecutter-style `{{ variable }}` in all
template files. Resolved at `maestro kit init` from:

```toml
# defaults.toml (kit-level)
[flutter]
organization = "com.example"
min_sdk_version = "21"
target_sdk_version = "34"
use_riverpod = true
use_go_router = true
```

```toml
# project brief (per-project)
app_name = "my-app"
description = "A Flutter application"
features = ["auth", "dashboard", "settings"]
```

**Post-generation hook**: `post_gen_project.py` runs `flutter pub get`,
initializes git, and optionally runs `flutter create` to regenerate
platform-specific boilerplate.

## Rationale

- Cookiecutter is the industry standard for project scaffolding (used by
  Django, FastAPI, Flask, and 1000+ community templates). No need to invent
  a new templating engine.
- Separate `defaults.toml` (kit-level constants) from the project brief
  (per-project variables) keeps shared config DRY while allowing per-project
  customization.
- Post-generation hooks automate setup steps that every Flutter project needs:
  dependency resolution and git initialization.

## Consequences

- Template files must escape any literal `{{ }}` that isn't a variable.
- The `{{project_slug}}` directory naming is a Cookiecutter convention вЂ”
  Maestro's template engine must support it.
- Platform-specific templates (android/, ios/, etc.) are regenerated by
  `flutter create` in the post-generation hook, so the template only needs
  the Dart source and platform config overrides.
- Adding a new feature template (e.g., `supabase_auth/`) means updating
  this ADR and the scaffold directory.
