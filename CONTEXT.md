# Flutter Adapter — Context

> Grilling session: 2026-05-29. Architecture decisions documented inline as they lock.

## Design system architecture

- **Q1 — HIG vs M3 agent split**: Three-agent pipeline. `spec-author` stays design-system-agnostic — generates abstract specs from primitives. Downstream: `hig-theme-builder` (Cupertino + Liquid Glass, iOS-native patterns) and `m3-theme-builder` (Material Design 3, tonal palettes, elevation). Matches the engineering-pack's own directory split (`_design-system/HIG/` vs `_design-system/M3/`). Each theme builder is a specialist for its design system — follows the specialist-agent principle from kit Q1. The seam between spec and theme is where the design system fork happens.
- **Q2 — Design-critic split**: Split into `hig-design-critic` + `m3-design-critic`. Each critic validates against its own design system's rules. HIG critic: Liquid Glass material correctness, Cupertino widget conformance, iOS-only deviations (toast→dialog). M3 critic: tonal palette accuracy, elevation levels, state layer application. Matches the split-builder pipeline — each builder's output feeds its own specialist critic.
- **Q3 — Rendering approach**: Hybrid. PlatformView native embeds (UiKitView/AndroidView) for visual-critical primitives where fidelity gap matters — Card with glassEffect, Button with native press motion, Dialog with native presentation. Standard Flutter widgets (Cupertino/Material) for the rest — TextField, Checkbox, layout, navigation. Agent knows which primitives warrant PlatformViews vs standard widgets. Follows the engineering-pack's existing PlatformView infrastructure for Button and Card.
- **Q4 — Widget-composer split + replica mode**: Split into `hig-widget-composer` + `m3-widget-composer` + `shadcn-widget-composer` + `replica-widget-composer`. HIG/M3 composers handle default mode — native widget skeleton (SwiftUI/Compose) skinned with design token styling. Shadcn composer uses shadcn_flutter for mobile — 30+ accessible components, fills the same role as a cross-platform component library. Replica composer handles `mode = replica` — pure Flutter pixel-perfect output, no native widgets, design is the single source of truth. Pipeline per platform: HIG (iOS native), M3 (Android native), shadcn (mobile, cross-platform), replica (all platforms, pure Flutter).
- **Q5 — Shadcn widget composer**: One `shadcn-widget-composer` for mobile Flutter. Uses shadcn_flutter — 30+ accessible components with built-in a11y, cross-platform on iOS and Android. No PlatformView embeds needed — shadcn_flutter components are pure Flutter with built-in accessibility. The symmetry: HIG (iOS native), M3 (Android native), shadcn (mobile cross-platform), replica (all platforms, design-as-source).
- **Q6 — Architecture pattern**: Stacked (production MVVM framework). Aligns with Flutter's official MVVM recommendation (Views + ViewModels + Repositories + Services + optional Domain layer). Built by FilledStacks (30+ production apps), 52.5k weekly downloads. Uses Provider + get_it for DI. Ships its own CLI (`stacked create app`) that scaffolds folder structure, reducing boilerplate the agent must generate. Agent's job: scaffold with Stacked CLI, fill ViewModels and Services, wire repositories. Production-proven, less code to generate, fewer agent mistakes.
- **Q7 — UI design skill split**: Three layered skills. `flutter-ui-design` (shared) — layout composition, responsive patterns, widget lifecycle, fundamentals both design systems need. `flutter-hig-design` (HIG extension) — Cupertino widgets, Liquid Glass tokens, iOS deviations, PlatformView/SwiftUI wrapping. `flutter-m3-design` (M3 extension) — Material widgets, tonal palettes, elevation, state layers, PlatformView/Compose wrapping. Mirrors the agent pipeline: shared spec-author → forked theme-builders. New design system = one new extension skill, not a full rewrite.
- **Q8 — Baseline mode**: Mode switch on existing HIG/M3/Shadcn composers, not new agents. Operator sets `tier = default | baseline`. Default: Liquid Glass, Expressive Alpha, enhanced materials (Tier 1). Baseline: standard Cupertino (`systemBackground`), standard M3 elevation, conservative materials (Tier 2/3). Same widget skeleton, different material tier. Tier parameter flows through the pipeline: spec-author → theme-builder(tier) → widget-composer(tier) → design-critic(tier). Maps to the engineering-pack's fidelity ladders and `data-glass-tier` runtime attribute.
- **Q9 — New skill flutter-platformview**: Dedicated skill for generating PlatformView native embed code. Covers UiKitView factory patterns (Swift), AndroidView factory patterns (Kotlin), MethodChannel bridge setup, view type registration in AppDelegate/AndroidManifest, hybrid composition vs virtual display mode selection, PlatformView lifecycle management, per-platform native build config. Complements flutter-platform-validation (tests PlatformViews) and flutter-ui-design (composes widgets). Three-skill chain: generate embed (platformview) → compose widgets (ui-design) → validate (platform-validation).

## Agent inventory

- **Q10 — Final agent count: 25**. 17 original agents → 3 split (theme-builder → 3, widget-composer → 4, design-critic → 2) → +6 agents (23). Plus `stacked-scaffolder` (24) and `idempotency-verifier` (25). Pipeline per platform: iOS → spec-author → hig-theme-builder → hig-widget-composer → hig-design-critic. Android → spec-author → m3-theme-builder → m3-widget-composer → m3-design-critic. Shadcn → spec-author → shadcn-theme-builder → shadcn-widget-composer (mobile cross-platform). Replica → spec-author → replica-widget-composer (no theme-builder, no design-critic). Every agent in application layer and below loads `flutter-stacked` skill for Stacked awareness. Per ADR-0001 split: Flutter adapter targets mobile only (iOS + Android). Web and desktop are owned by the dioxus-adapter.

## Stacked × DDD integration

- **Q11 — Stacked within DDD bounded contexts**: Feature-first layout where each DDD bounded context becomes a feature folder. Inside: `ui/views/` (Stacked Views), `ui/viewmodels/` (Stacked ViewModels with BaseViewModel), `domain/` (DDD entities, value objects), `data/repositories/` (Stacked-compatible, get_it registered), `data/services/` (Stacked services), `application/` (DDD commands/queries). Stacked's CLI scaffolds the app skeleton; widget-composers fill ViewModels and Views within existing bounded contexts. New shared skill `flutter-stacked` teaches all agents ViewModel patterns, service registration, routing annotations.

## DDD swapability

- **Q12 — UI layer swapability**: The DDD architecture allows swapping the entire presentation layer without touching domain or application layers. Domain is pure Dart — no Flutter, no Stacked. Application layer depends on domain interfaces, not UI. Two levels of swap: (1) UI framework swap within flutter-adapter (Stacked → BLoC → plain MVVM), domain survives. (2) Stack swap across adapters (flutter-adapter → dioxus-adapter), same bounded contexts, different language, same DDD structure.

## Determinism model

- **Q13 — Eight-layer determinism model**: (1) Deterministic scaffolding — `stacked create app` CLI, zero AI. (2) Schema-constrained generation — structured design-analysis.json → agent prompts. (3) Adapter table mapping — intent/motion/visual tables as deterministic lookup. (4) Golden-spec validation — verifier agent runs probe-runner: design_bundle vs app_bundle with skeleton_diff. (5) Idempotent pipeline — stable sorts, no UUIDs, no timestamps. (6) Snapshot-idempotency gate — verifier agent runs pipeline twice, diffs bundles via probe-runner. (7) Compiled primitives — agent generates primitive code once, probe-runner certifies, output stored as compiled artifact. Same input → compiled output reused, zero LLM tokens. Change input → regenerate. No hand-coded templates. (8) Canonical output formatter — build script normalizes imports, formatting, file ordering.
